Project Overview

This project is a production-grade distributed job scheduling platform capable of reliably executing asynchronous background jobs across multiple workers. It directly addresses every requirement in the Codity.AI intern assignment specification.

The platform is modeled after real-world systems like Celery, BullMQ, and AWS SQS — with a focus on correctness (atomic claiming, idempotency), observability (real-time dashboard, execution logs), and reliability (retry strategies, dead-letter queue, self-healing workers).

Core Features Implemented

🔐
Authentication & Multi-tenancy
JWT auth with register/login/refresh. Full Org → Project → Queue → Job hierarchy with RBAC (Owner/Admin/Member/Viewer).
⚡
5 Job Types
Immediate, Delayed, Scheduled (datetime), Recurring (cron), and Batch submissions — all via REST APIs.
🔒
Atomic Job Claiming
Zero duplicate executions guaranteed via PostgreSQL's SELECT FOR UPDATE SKIP LOCKED — no application-level locks needed.
🔄
Retry Strategies
Fixed, Linear, and Exponential backoff with optional jitter. Configurable per-queue via Retry Policies.
💀
Dead Letter Queue
Permanently failed jobs land in the DLQ with full error context. Human operators can retry or discard them from the dashboard.
💓
Worker Heartbeats
Workers send heartbeats every 5s. Stale workers (30s timeout) are auto-detected and their jobs are recovered and re-queued.
📊
Real-Time Dashboard
React dashboard with live WebSocket updates — job state changes, queue health, worker utilization, and throughput charts appear instantly.
🌐
REST APIs
25+ endpoints covering all resources. Auto-generated Swagger UI available at /docs.
🔑
API Keys
Project-scoped API keys for programmatic access. Keys are stored as bcrypt hashes — only the prefix is stored in plaintext.
🔗
Workflow Dependencies
Jobs can declare dependencies on other jobs — a DAG-based chain where Job B only runs after Job A completes.
⏸️
Queue Control
Queues can be paused and resumed on demand. Paused queues stop accepting new job claims without discarding pending jobs.
📈
Metrics & Statistics
Per-queue stats snapshots saved on a schedule. Throughput history, success rate, avg execution time surfaced in the metrics page.

Key Engineering Decisions

1. Atomic Job Claiming — SELECT FOR UPDATE SKIP LOCKED

This is the most critical design decision. When multiple workers poll simultaneously for the same queue, they will all see the same pending jobs. A naive implementation would have workers racing to claim the same job — resulting in duplicate execution.

PostgreSQL's FOR UPDATE SKIP LOCKED acquires a row-level lock atomically within the transaction. Any row already locked by another transaction is skipped — not waited on. This guarantees exactly-once claiming at the database layer. No application-level distributed locks, no Redis needed.

SELECT id, handler, payload FROM jobs
WHERE queue_id = $1 AND status = 'pending'
  AND deleted_at IS NULL
ORDER BY priority DESC, created_at ASC
LIMIT $2
FOR UPDATE SKIP LOCKED  -- ← The key

2. Retry Strategies

StrategyDelay FormulaUse Case
Fixeddelay = initialSMS/email retries with fixed cooldown
Lineardelay = initial × attemptRate-limited external APIs
Exponentialdelay = initial × multiplier^(n-1)Database overload, transient failures
+ Jitterdelay × (0.5 + rand × 0.5)Prevents thundering herd on retry wave

3. Self-Healing Workers

Workers that crash (power loss, OOM kill, segfault) without sending a shutdown signal will be detected automatically. The scheduler loop runs on every worker and periodically checks:

SELECT id FROM workers
WHERE status IN ('active','idle')
  AND last_heartbeat_at < NOW() - INTERVAL '30 seconds'

Detected stale workers are marked offline. Their claimed or running jobs are set back to pending and re-queued — no human intervention required.

4. Year-Partitioned Tables

The job_executions and job_logs tables are range-partitioned by year. Benefits:

  • Purging old data is instant — just DROP TABLE job_executions_2024
  • Active queries only scan the current year's partition
  • PostgreSQL's constraint exclusion automatically skips irrelevant partitions

5. Idempotency Keys

Clients can submit jobs with an idempotency_key. A partial unique index prevents duplicate rows:

CREATE UNIQUE INDEX idx_jobs_idempotency ON jobs(queue_id, idempotency_key)
WHERE idempotency_key IS NOT NULL AND deleted_at IS NULL;

If a client submits the same job twice (due to a network retry), the second request returns the original job record — no duplicate execution.

Setup & Running the Project

Prerequisites

  • Python 3.10+ (tested with 3.12)
  • Node.js 18+ and npm
  • PostgreSQL 14+ (tested with 17)

Step 1 — Database Setup

# Create the database
psql -U postgres -c "CREATE DATABASE codityai;"

# Apply the full schema (20 tables, triggers, indexes)
psql -U postgres -d codityai -f backend/migrations/001_initial.sql

Step 2 — Backend Configuration

Edit backend/.env:

DATABASE_URL="postgresql://postgres:<password>@localhost:5432/codityai"
SECRET_KEY="your-256-bit-secret-key"
WORKER_MAX_CONCURRENCY=10
WORKER_POLL_INTERVAL=1.0
WORKER_HEARTBEAT_INTERVAL=5.0

Step 3 — Start the API Server

# Virtual environment is pre-created at backend/venv/
cd backend
.\venv\Scripts\activate          # Windows
# source venv/bin/activate       # Mac/Linux

uvicorn app.main:app --reload --port 8000

→ API available at http://localhost:8000
→ Swagger UI at http://localhost:8000/docs

Step 4 — Start the Worker Engine

# Open a NEW terminal (with venv activated)
cd backend
python -m app.worker.main

The worker will register itself in the database and begin polling for jobs.

Step 5 — Start the Frontend Dashboard

cd frontend
npm install    # if first time
npm run dev

→ Dashboard at http://localhost:5173
